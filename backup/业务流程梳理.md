## 简介
做链上数据采集和分解析的
上游：链上数据采集与解析    中游：规则引擎解析     下游：告警、跟单、消息推送
技术栈： Kafka / Redis / ClickHouse / InfluxDB / Mongo / Postgres
主体业务：从节点 RPC 拉取区块/日志，解析 DEX 交易与资金流，产出结构化事件进入 Kafka/存储，并通过规则引擎生成告警，最终分发至 Redis 队列与终端推送

1. 数据来源：一条是 Kafka（区块或交易聚合后的数据流），另一条是 Redis PubSub（更实时的 swap 事件流）。在服务启动时会分别起 Kafka consumer 和 Redis consumer
2. 处理流水线：两路 consumer 都不是直接写业务逻辑，而是注册一串 Processor：典型是解码 Processor把原始消息转成内部结构，再用过滤/去重 Processor做降噪。它们最终都把结果写到同一个 ChannelSink，相当于把多来源数据统一进一个通道。然后 MergeChannelProducer 会从 ChannelSink 取出消息，序列化为 ChanMsg 放进 mergeChan，这样后续就能用同一个 ConsumeChanMsgFactory 进行统一消费和后处理
3. 规则查询：mergeChan 的消费端会通过 ProcessorRegistry 顺序执行后续 Processor。第一段通常是 MergeSwapRecordsProcessor：它把 Kafka/Redis 的消息反序列化成统一格式，抽取并合并 swap，最终产出 []SingleSwapResult（统一的交易摘要，包含 maker、净流入流出、协议等）。接下来进入规则引擎：我这边的实现是接口化的 MonitorRule，策略来自用户配置加载器（MongoDB 定时同步到内存）。检查是分层的：先跑链/时间段等轻量规则，再跑 swap 里能直接取到的交易额、协议等规则，最后才跑像“钱包组全买”这种需要统计/聚合甚至依赖外部数据的重规则。这样可以在高吞吐下尽早过滤，降低重规则成本。
4. 告警/跟单触发逻辑
规则命中后 Processor 会返回 KafkaMessageGroup 作为输出：
告警链路：由 ChanMsgFactory 的发送协程统一把结果发到 Kafka；如果配置打开，还会用自定义 send 同步把告警推到 Redis 队列，给其他模块消费
跟单链路：命中后生成跟单信号/任务，通过配置的 Sink（例如落到队列/Mongo）沉淀，后续执行模块消费并完成通知。
